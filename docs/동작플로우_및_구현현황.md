# 감사조서 자동화 — 동작 플로우 & 구현 현황

> 이 문서는 시스템이 **무엇을, 어떻게 동작하게 만들었는지**를 글로 설명합니다.
> 피감회사 데이터·법인 양식은 공개하지 않으므로(데이터 정책), 코드 로직과 처리 흐름만 기술합니다.
> 화면에 보이는 실제 수치·회사명·양식 파일은 포함하지 않습니다.

---

## 0. 한 줄 요약

회사마다 제각각인 **비정형 입력(정산표·원장·조회서·분개장)을 정규화**하고, 그 정형 데이터를
**회계법인 완성본 조서 양식에 무손실로 매핑·대사**해 초안을 생성한다. 수치는 100% 규칙 기반으로
처리(환각 차단)하고, 회계사가 검토·보완해 완성한다.

배포는 **실행기 2개**로 동작하며, 한 폴더 안에서 **여러 회사를 격리 관리**한다.

---

## 1. 사용자 동작 플로우 (인차지 관점)

```
① 작업생성기.exe          회사명 입력 → 작업/{회사}/ 폴더 자동 생성
   └ 입력자료·변환자료·참고자료·출력조서 4폴더 + 입력 안내문 + 환율/상각 템플릿 동봉

② 입력자료에 엑셀 투입      파일명을 역할 한 단어로(정산표·조회서·잔액·분개장·발송CS·관리대장)
   └ 회사마다 원본 이름이 달라도, 라벨/토큰으로 자동 인식

③ 조서생성기.exe          회사 선택 → 회사정보(기준일·작성자·검토자) 1회 입력(이후 저장·재사용)
   └ 조서 메뉴에서 선택: 하나만('4') / 여러 개('1,3,4') / 전체('A')
   └ 선택 조서를 의존성 순서로 생성

④ 출력조서/               완성 초안 + 실행 리포트(.txt)
   └ 무엇이 입력 문제이고 무엇이 양식 문제인지 구분해 보고
```

**핵심 UX 설계**
- 회사 = 폴더 하나. 회사를 바꿔도 입력·결과·캐시가 섞이지 않는다.
- 회사정보는 한 번만 입력하면 저장되어, 같은 회사의 다른 조서를 만들 때 재입력이 없다.
- 경로·플래그를 몰라도 된다. 파일 위치만 맞추면 나머지는 자동.

## 2. 내부 처리 파이프라인 (STEP 1~5)

### STEP 1. 데이터 추출·정형화 — "비정형 → 이상적 양식 중간다리"

회사마다 다른 입력을 표준 형태로 수렴시킨다.
- **헤더 매핑 3겹 안전망**
  - C-1차(오프라인·무료): 정확 동의어 매칭 + 부분일치(fuzzy) 폴백 — 못 찾은 *필수* 필드만 보강.
  - C-2차(선택·마스킹 LLM): C-1차로도 못 찾은 헤더 행만 좁게 질의. **셀 값은 전부 마스킹**(자릿수/
    길이/빈칸 기호만, 실값 0 누출)하고 헤더 라벨만 평문 전송. 해소된 매핑은 학습 캐시에 저장(복리).
  - 키 없으면 LLM 비활성 = 100% 오프라인.
- **라벨 우선 + 다중 헤더행 스캔**: 라벨이 어느 행(헤더/서브헤더)에 올지 회사마다 달라, 첫 행만 보지
  않고 서브헤더행까지 함께 본다(조용한 누락 방지). 위치 오프셋은 라벨이 없는 변형 전용 폴백.
- **소스 해시 기반 캐시**: 입력 파일이 안 바뀌면 재파싱하지 않고, 바뀌면 해시가 달라져 자동 재생성.
  회사별로 격리되어 충돌·stale을 원천 차단.
- **독립 변환 도구로도 제공**: 분개장→이상적 분개장 / 잔액명세서(원장)→이상적 잔액명세서 변환은
  **폴더 자동감지 오프라인 도구**로 단독 실행할 수도 있다(더블클릭·API 불필요). 파이프라인과 같은
  파서를 재사용하고, `.xls/.xlsx/.xlsm`·다중시트·이중계정·거래처블록 변형을 흡수하며 차대 균형을
  점검한다.

### STEP 2. 조서 초안 생성 — 총괄표 우선 → 개별조서 동적 구성

- **총괄표 우선 + 총괄표=드라이버**: 정산표에서 16개 조서 총괄표를 **먼저** 일괄 생성하고(+완전성
  대사), 그 총괄표가 개별 상세조서를 이끈다. 개별조서는 총괄표의 값(수식 링크)·범위·표 크기와
  연동돼 **동적 구성**된다 — 넓은 표준 양식에서 그 회사에 불필요한 부분을 빼(subtractive) 필요한
  만큼만 남긴다. 개별 상세 생성기가 있는 계정은 그 결과가 더 완전한 최종본으로 총괄표를 덮어쓴다.
- **이상적 자료 조합**: 개별조서가 필요로 하는 재료(분개장·잔액원장 등)는 STEP 1에서 이상적
  양식으로 변환된 것을 조서별로 꺼내 조립한다. 예) BBDD·G는 이상적 분개장, G는 관리대장,
  A-0/A-1은 잔액·조회서.
- **계정유형별 분기**: 같은 양식 틀이라도 채우는 로직이 계정 성격에 따라 다르다.
  - 잔액형(현금·차입금): 기초 → 증감 → 기말 중심.
  - 변동분해형(유형자산): 거래를 취득·처분·감가상각 원인별로 분류.
- **완성본 무손실 이식(graft)**: 회계법인 완성본은 보조시트에 양식컨트롤·매크로·도형이 가득하다.
  통째로 열고 저장하면 이들이 소실되므로, **총괄표 한 장만 가볍게 추출→채움→그 시트만 zip/XML
  레벨로 이식**한다. 나머지 시트·컨트롤·매크로는 바이트 그대로 보존된다. 시트별로 연쇄 가능.

### STEP 3. 양식 무결성 검증 — "시각 확인의 프로그래밍 대체"

엑셀 수식은 값만 계산해 시각적 양식 붕괴(테두리 소실·소계 어긋남·행 밀림·헤더 색 변질)를 못 잡는다.
생성 직후 구조 불변식을 검사하는 게이트를 통과시킨다: 격자(테두리)·소계 SUM 범위 일치·열 잘림·
제목/헤더 존재·헤더 채움색 대조 등. **0 issue 확인 후에야 완료**로 본다.

### STEP 4. 서술 생성 (선택) — Test 목적·방법·결론 초안 (Claude API)

### STEP 5. 회계사 보완·완성 — 초안을 검토하고 추가 절차를 수행해 완성

## 3. 구현된 조서 (실적)

| 조서 | 입력 | 내용 |
|---|---|---|
| **조서생성** | 정산표 | 16개 조서 **총괄표 일괄** 생성 + **완전성 대사**(어느 조서에도 안 담긴 미매핑 계정·수정분개를 리포트) |
| **A-0 현금** | 정산표·조회서·잔액 | 총괄표 + 보험가입현황 + 장·단기금융 평가대사 |
| **A-1 금융기관 조회** | 발송CS·조회서·잔액 | A100 완전성체크 + A200 조회서 대사 + A300 기타조회(퇴직연금·대출·담보) |
| **BBDD 차입금** | 정산표·조회서·잔액·분개장 | 조회박스·담보표·이자율별 상세 |
| **G 유무형자산** | 정산표·관리대장·분개장 | 변동분해 롤포워드 재계산(정률/정액 추론) |
| **R·S 판관비·기타손익** | 정산표 | 증감분석(중요성 임계 기반)·주석검토 보조시트 |
| **D·CC 기타자산·부채 상세** | 정산표·잔액 | 거래처별 명세 동적 생성(총괄표 없으면 자체 선생성) |

> 16개 총괄표 중 개별 상세 생성기가 없는 계정(매출채권·매입채무·재고·매출원가·매출·자본·인건비·
> 퇴직급여·매도증권 등)은 **조서생성이 유일한 자동화**이고, 완전성 대사는 조서생성만 수행한다.
> 개별 상세 생성기가 있는 계정은 그 결과가 더 완전한 최종본으로 총괄표를 덮어쓴다.

## 4. 견고성·신뢰성 설계 (감사 적합성)

- **부분 실패 허용**: 양식 하나가 안 맞아도 전체가 멈추지 않는다. 실패한 부분만 비우고 나머지는
  생성하며, **진단 리포트**로 입력 문제와 양식 문제를 구분해 보고한다(LLM이 못 보는 결과를 사용자
  눈으로 대체).
- **조서 간 결합 최소화**: 7개 조서가 서로 독립이라 단독/임의조합으로 생성 가능. 실질 의존은
  "총괄표 → 일부 상세" 한 줄뿐이고, 그마저 없으면 자체 선생성하도록 독립화.
- **회사별 캐시 격리**: 회사를 바꿔도 중간 캐시가 섞이지 않는다.
- **변형 대응(예시)**: 외부확인 항목이 표준 위치에 없는 변형에 대응한다. 예) 보험 해약환급금이
  표준 시트에 없고 다른 시트/섹션에 있는 회사 → **'해약환급금' 열을 시트·위치 불문 라벨로 스캔**해
  장기금융상품으로 끌어온다(위치 하드코딩 아님). 기존 표준 파서는 불변이라 기존 회사 무회귀.

## 5. 규칙 기반 vs LLM 경계 (환각 차단의 핵심)

| | 규칙 기반(Python) | LLM(선택) |
|---|---|---|
| 담당 | 셀 매핑·합계·잔액 대사·만기/금액 분류 — **모든 수치** | 비정형 헤더 분류(미해결 필드만)·Test 서술 초안 |
| 보안 | — | API 키 외부 보관, egress 전 동의·전송내용 표시, 셀 값 마스킹 |

수치 판단은 절대 LLM에 맡기지 않는다. LLM은 비정형 분류와 서술 초안에만 제한적으로 쓰이며,
없어도(오프라인) 핵심 기능은 그대로 동작한다.

---

## 6. 데이터 정책

피감회사 데이터·법인 양식은 보호 대상이다. 공개 저장소에는 **처리 로직(코드)만** 포함하고, 실무
데이터·합성 데이터·조서 양식·배포물은 모두 제외한다. API 키는 환경 변수로만 관리한다. 커밋
게이트(pre-commit 훅)가 비공개 경로·엑셀·식별번호·회사명 유입을 자동 차단한다.

## 7. 현재 단계

실제 감사 업무에 적용해 목표한 조서 세트를 **완성·제출**했고, 의도한 대로 동작함을 확인했다.
"정형 반복 작업 자동화"라는 본래 목표는 달성했으며, 적용 과정에서 **몇 가지 보완점과 확장 여지**를
확인해 다음 단계로 이어가고 있다.

- **완수**: 7개 조서 세트 + 2-실행기(작업생성기·조서생성기) + 회사 워크스페이스 격리 + 변형 대응
  (해약환급금 위치불문 스캔)까지 동작 검증하고 실제 산출물로 활용.
- **보완·확장 후보**: 급여(인건비) 등 미구현 계정 매핑 확대, 회사별 변형 양식 대응 케이스 축적,
  임의(중간)감사용 간이 매핑 적용.
