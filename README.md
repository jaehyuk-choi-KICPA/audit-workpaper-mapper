# 감사조서 자동화 시스템 (Audit Workpaper Automation)

비상장 회사 외부감사에서 반복되는 조서 작성 작업을 자동화하는 프로젝트입니다. 정산표·분개장·조회서·원장 데이터를 정해진 조서 양식에 매핑·대사하는 **규칙 기반 처리**와, 비정형 입력의 분류·서술이 필요한 부분만 **LLM(Claude API)**으로 처리하는 하이브리드 구조로 설계했습니다.

회계 실무(KICPA)를 배경으로, **2026년 6월 중순~7월 초 회계법인 재직 중 용역성 업무로 개발한** 실무 자동화 프로그램입니다. 이 **공개 저장소에는 처리 로직(코드)만** 포함되며, 피감회사 데이터·조서 양식·배포 바이너리는 포함되어 있지 않습니다.

> 데이터 없이 서술한 **동작 플로우·구현 현황** 문서: [docs/동작플로우_및_구현현황.md](docs/동작플로우_및_구현현황.md)

## 왜 만드는가

비상장 회사 감사는 계정별 절차가 상당 부분 정형화되어 있고 반복적입니다. 회계사가 정산표·원장의 수치를 정해진 조서 양식에 수기로 옮기고 외부 조회서와 대사하는 노가다성 작업이 큰 비중을 차지합니다. 이 정형 반복 작업을 자동화해 회계사가 판단·검토에 집중하게 하는 것이 목표입니다.

프로젝트의 두 축은 **(1) 회사마다 제각각인 비정형 입력의 정형화**와 **(2) 정형 데이터의 노가다성 조서 매핑 자동화**입니다.

## 무엇을 자동화했나

| 구성요소 | 입력 | 하는 일 |
|---|---|---|
| **총괄표 일괄 생성** | 정산표 | 16개 조서의 총괄표(lead schedule)를 한 번에 생성하고, **완전성 대사**로 어느 조서에도 안 담긴 미매핑 계정·수정분개를 리포트 |
| **개별 상세조서** | 조서별 상이 | A-0 현금 · A-1 금융기관 조회 · BBDD 차입금 · G 유무형자산 · R·S 판관비/기타손익 · D·CC 기타자산/부채 상세 |
| **독립 변환 도구 2종** (오프라인) | 분개장 / 잔액명세서 | 회사마다 다른 원본을 **이상적(표준) 양식**으로 변환 — API 불필요, 폴더에 두고 더블클릭 |
| **2-실행기 배포** | — | 작업생성기(회사 워크스페이스 생성) + 조서생성기(조서를 하나만/여러 개/전체 선택 생성) |

## 핵심 아키텍처 — 총괄표가 개별조서를 이끄는 구조

중심 설계는 **총괄표를 먼저 채우고, 그 총괄표가 개별 상세조서를 이끄는** 흐름입니다.

```
① 총괄표 일괄 생성   정산표 → 16개 조서 총괄표 + 완전성 대사(미매핑 계정·수정분개 리포트)
② 총괄표 = 드라이버   개별조서가 총괄표의 값·범위·표 크기와 연동돼 동적 구성(subtractive)
③ 이상적 자료 조합   분개장·잔액원장을 표준 양식으로 변환해 조서별로 조립
④ 완성본 graft      총괄표 시트만 무손실 이식(양식컨트롤·매크로·도형 보존)
⑤ 양식 무결성 검증   격자·소계·열·헤더색 구조 불변식 게이트
```

생성 결과는 최종 완성본이 아니라 **초안**입니다. 회계사가 검토·보완해 완성합니다.
단계별 상세 동작은 [docs/동작플로우_및_구현현황.md](docs/동작플로우_및_구현현황.md)에 별도로 정리했습니다.

### 계정 유형별 처리 분기 (routing)

조서 양식(틀)은 표준화되어 있으나, 같은 틀에 데이터를 채우는 계산·매핑 로직은 계정 성격에 따라 다릅니다.

- **잔액형** (현금·차입금): 기초 → 증감 → 기말 잔액 중심으로 채움
- **변동분해형** (유형자산): 거래를 취득·처분·감가상각 등 원인별로 분류해 채움

처음부터 모든 분기를 구현하지 않고, 계정을 하나씩 추가하며 점진적으로 확장합니다.

## 독립 변환 도구 (standalone)

"비정형 입력의 정형화" 절반을 단독으로 수행하는 오프라인 도구 두 개를 별도로 만들었습니다. 폴더(하위 포함)에서 대상 파일을 자동 감지해 각각 표준 양식으로 변환하고, 결과를 `변환결과/`에 저장합니다.

- **분개장 → 이상적 분개장** — 날짜·번호·계정과목·거래처·적요·차변·대변 표준표로 변환하고 차변합=대변합 균형을 점검. (`src/easy_run_convert_journal.py`)
- **잔액명세서(거래처원장) → 이상적 잔액명세서** — 계정과목·거래처별 기초·기말 표준표로 변환. (`src/easy_run_convert_ledger.py`)

여러 시트 중 데이터 시트 자동 선택, `.xls/.xlsx/.xlsm` 인식, 다중시트·이중계정·거래처블록 등 변형 양식을 흡수합니다. **API 키가 없어도 100% 동작**합니다(오프라인).

## 설계 원칙

### 1. 규칙 기반과 LLM의 역할 분리

| 처리 방식 | 담당 작업 |
|---|---|
| 규칙 기반 (Python·openpyxl) | 셀 매핑, 합계 계산, 잔액 대사, 만기·금액 기준 분류 — **모든 수치** |
| LLM (Claude API, 선택) | 비정형 입력의 헤더/계정 분류(미해결 필드 한정), Test 서술 초안 |

정형 매핑·수치 대사는 반드시 규칙 기반으로 처리해 환각(hallucination)을 차단합니다. LLM에는 수치 판단을 맡기지 않고, 비정형 분류와 서술 초안에만 제한적으로 사용합니다. API 키가 없으면 LLM은 비활성화되고 핵심 기능은 그대로 동작합니다.

### 2. Workflow 방식 (자율 에이전트 아님)

감사 절차는 결정론적이므로 처리 순서를 코드에 명시적으로 정의합니다. LLM이 전체 흐름을 자율 결정하는 에이전트 구조 대신, 단계가 통제·검증 가능한 워크플로우로 설계했습니다. 감사는 일관성·추적가능성이 핵심이라 의도적으로 이 방향을 택했습니다.

### 3. 검증과 사람 검토 (Human-in-the-loop)

- 시스템 출력물은 **초안**입니다. 재분류 등 판단이 걸린 작업은 LLM이 "제안"만 하고 회계사가 확정합니다.
- 생성 직후 합계 대사·누락 여부와 **양식(테두리·소계·열·헤더색) 무결성**을 자동 검증하는 게이트를 거칩니다.

### 4. 견고성 (부분 실패 허용)

회사마다 입력 양식이 조금씩 달라 파서가 안 맞을 수 있습니다. 양식 하나가 안 맞아도 조서 전체가 출력 안 되면 안 되므로, 모든 파서·생성기 호출을 안전 래핑해 **한 부분이 실패해도 나머지는 생성**하고, 무엇이 입력 문제이고 무엇이 양식 문제인지 구분한 진단 리포트를 남깁니다. 헤더 매핑은 오프라인 동의어·부분일치(C-1차)를 우선하고, 미해결 필수 필드만 선택적으로 마스킹 LLM(C-2차, 셀 값 전부 마스킹)에 질의합니다.

## 사용 인터페이스 / 배포

배포 패키지는 **두 개의 실행기**로 동작하며, 한 폴더 안에서 여러 회사를 **회사별 워크스페이스**로 격리 관리합니다.

```
자동화프로그램/
 ├─ 작업생성기.exe   ① 회사명 입력 → 작업/{회사}/ 4폴더 생성(+입력·참고 안내문 동봉)
 ├─ 조서생성기.exe   ② 회사 선택 → 조서 메뉴(하나만 '4' / 여러 개 '1,3,4' / 전체 'A')로 생성
 ├─ 양식자료/        조서 템플릿 (회사 무관 공유)
 └─ 작업/{회사}/     회사명 = 폴더명(자동 인식)
     ├─ 입력자료/    원본 엑셀 투입 (파일명을 역할 한 단어로: 정산표·조회서·잔액·분개장·관리대장…)
     ├─ 변환자료/    이상적 양식으로 변환·캐시된 중간물 (소스 해시 기반, 회사별 격리)
     ├─ 참고자료/    환율·내용연수 등 참조 (회사별)
     └─ 출력조서/    생성된 조서 + 실행 리포트(.txt)
```

- 회사·작성자·기준일은 **한 번만** 입력하면 저장되어, 같은 회사의 다른 조서를 생성할 때 재입력이 없습니다.
- 조서들은 서로 독립이라 단독·임의조합으로 생성할 수 있고, 실질 의존("총괄표 → 일부 상세")은 실행기가 순서로 자동 처리합니다.
- 회사를 바꿔도 입력·결과·캐시가 섞이지 않습니다.

## 기술 스택

- Python 3.11 / openpyxl (엑셀 입출력·매핑·대사)
- anthropic (Claude API, 서술·비정형 분류) · python-dotenv (API 키 관리)
- PyInstaller (배포 실행기 빌드)

## 데이터 정책 (Compliance)

피감회사 데이터와 법인 양식은 모두 보호 대상이므로, 데이터 거버넌스를 설계에 반영했습니다.

- 이 공개 저장소에는 처리 로직(코드)만 포함합니다. 실무 데이터는 물론, 합성 데이터와 조서 양식도 제외합니다(합성 데이터라도 양식 구조가 법인 자산에 해당). 배포 실행 바이너리(`.exe`)도 포함하지 않습니다.
- 데이터·양식·배포물은 비공개 폴더(`_internal/`)와 작업폴더에 두며 버전 관리에서 제외합니다.
- API 키 등 민감 정보는 환경 변수로만 관리하고 저장소에 포함하지 않습니다.
- 커밋 게이트(pre-commit 훅)가 비공개 경로·엑셀·식별번호·회사명 유입을 자동 차단합니다.

## 프로젝트 구조

```
.
├── src/                         # 처리 로직 (공개)
│   ├── extractors/              #   데이터 파싱 (정산표·원장·조회서·분개장 → 이상적 양식)
│   ├── generators/              #   조서 생성 (계정별 분기 · 완성본 graft · 총괄표 드라이버)
│   ├── workspace.py             #   회사별 작업폴더 공통 계층
│   ├── pipeline.py              #   STEP 워크플로우
│   ├── validator.py             #   합계 대사·양식 무결성 게이트
│   ├── easy_run_all.py          #   조서생성기 (통합 실행기: 하나만/여러 개/전체)
│   ├── easy_run_setup.py        #   작업생성기 (회사 워크스페이스 생성)
│   └── easy_run_convert_*.py    #   독립 변환 도구 (분개장·잔액원장 → 이상적 양식)
├── docs/                        # 동작 플로우·구현 현황 (데이터 없는 공개용 서술)
├── _internal/                   # 비공개 (버전 관리 제외): 데이터·양식·config·배포물
├── CLAUDE.md                    # 프로젝트 컨텍스트(설계 규칙)
└── README.md
```

공개되는 것은 `src/`의 처리 로직과 `docs/`의 서술뿐입니다.
