# 감사조서 자동화 시스템 — 프로젝트 컨텍스트

## 프로젝트 개요

비상장 회사 외부감사에서 반복적으로 수행되는 조서 작성 작업을 자동화하는 프로젝트.
정산표·분개장·조회서 데이터를 정해진 조서 양식에 매핑·대사하는 규칙 기반 처리와, 판단·서술이 필요한 부분만 Claude API로 처리하는 **하이브리드 구조**.

배경: KICPA 회계 실무 + 개발 학습을 병행하며 진행 중인 프로젝트.

---

## 설계 원칙 (변경 시 반드시 확인)

### 1. 규칙 기반 vs LLM 역할 분리

| 처리 방식 | 담당 작업 |
|---|---|
| 규칙 기반 (Python · openpyxl) | 셀 매핑, 합계 계산, 잔액 대사, 만기·금액 기준 분류 |
| LLM (Claude API) | 비정형 입력의 헤더/계정 분류, Test 서술 생성 |

- **환각(hallucination) 방지**: 정형 매핑·수치 대사는 반드시 규칙 기반으로 처리. LLM에 수치 판단을 맡기지 않는다.
- LLM은 서술 초안 생성과 비정형 분류에만 사용.

### 2. 워크플로우 방식 (자율 에이전트 아님)

- 처리 단계(STEP) 순서와 분기는 코드에 **명시적으로 정의**.
- LLM이 전체 처리 흐름을 자율 결정하는 에이전트 구조는 금지. 감사는 일관성·추적가능성이 핵심이므로 단계가 통제·검증 가능한 워크플로우를 유지한다.
- 단, 특정 단계 내에서 범위가 고정된 bounded tool use는 허용 (예: 서술 생성 단계에서 잔액 조회 후 서술에 반영).

### 3. Human-in-the-loop

- 시스템 출력물은 **초안**이다. 최종 완성본이 아님.
- 재분류 등 판단이 걸린 작업은 LLM이 "제안"만 하고, 회계사가 확정한다.
- 조서 생성 후 합계 대사·누락 여부를 자동 검증한다.

### 4. 양식 무결성 (필수 — 위반 금지)

조서에서 **양식(테두리·격자·서식) 무결성은 수치만큼 중요**하다. 양식이 무너지면 조서 신뢰성이
훼손된다. 모든 데이터-채우기 작업은 다음 두 단계를 **반드시** 거친다:

1. **기존 양식 보존**: 템플릿에 미리 그려진 격자를 파괴하지 않는다.
   - **고정 격자 시트**(예: Control Sheet) → `base.clear_values`(값만 비움, 테두리 보존),
     격자 없는 행에만 도너 스타일(`capture_row_style`/`apply_row_style`) 복사.
   - **동적 표**(예: A200, 행수 가변) → `clear_region`+`render_*`로 도너 스타일 복사하며 재구성.
   - **헤더 보존 (필수 — 반복 사고 예방)**: 각 시트의 헤더 행은 **값·채움색(fill)을 절대 잃거나 바꾸지
     말 것**. 특히 ⚠️ **헤더 색을 임의의 새 색(파란색 등)으로 바꾸지 않는다** — 헤더엔 새 fill을 입히지
     말고 **템플릿 원래 색을 그대로** 둔다. `clear_region`은 fill을 빈 PatternFill로 리셋하므로,
     헤더가 클리어 범위에 들면 반드시 `capture_row`(값+스타일+fill)로 떠놨다가 `stamp_row`로 **원래
     fill째 재현**한다(헤더는 가급적 클리어 범위 **밖**에 두는 게 안전). 도너는 반드시 **그 시트의 실제
     헤더 행**에서 캡처 — 데이터 행/다른 시트 도너를 헤더에 적용 금지(엉뚱한 색=파란색 변질의 원인).
2. **사후 종합 검사 = 시각 확인의 프로그래밍 대체 (필수)**: LLM은 생성된 엑셀을 **눈으로 볼 수 없으므로**,
   '완료' 전에 반드시 종합 양식검증을 실행·보고한다 — `validate_grid`(테두리) + `check_column_fit`
   (열 잘림) + `check_cells_present`(제목·헤더 존재) + `check_header_fill`(헤더 채움색 소실·변질[파란색]을
   템플릿과 대조) (+A200 `validate_a200_form`). 예방으로
   `fit_column_widths`(열 너비 보정)도 적용. 파이프라인 `_check_form_integrity` HOOK이 자동 실행.
   **0 issue 확인 후에야 완료 보고** (생략하면 사용자가 매번 크로스체크 → 비효율 반복).

→ **새 시트/조서를 추가할 때 이 두 단계를 빼먹지 말 것.** 기존 base 도구·validator를 재사용한다.
(근거: Control Sheet에서 `clear_region` 오용으로 격자 붕괴 + 주소열 너무 좁음·
순서·회사명 누락이 시각으로만 발견된 사건. 검증이 테두리만 봐서 너비·제목을 놓침 → 항목 확장.)

### 5. 견고성 + 진단 (절대 전체실패 금지 — 프로젝트 최우선)

회사마다 입력 양식이 조금씩 달라 파서/생성기가 안 맞을 수 있다. **양식 하나가 안 맞아도 조서
전체가 출력 안 되면 안 된다.** 따라서:
- **부분 실패 허용**: 모든 파서·생성기 호출은 `pipeline._safe_parse`/`_safe_gen`로 감싸, 한 부분이
  실패하면 그 부분만 비우고 **나머지는 생성**한다(항상 출력 파일 생성). 파서는 "데이터/시트 없음"을
  예외 대신 **빈 결과**로 반환.
- **진단 리포트**: `build_a1`은 `RunReport`(입력/생성/출력/확인필요)를 반환. **무엇이 입력값 문제이고
  무엇이 출력 양식 문제인지** 구분해 보고. `easy_run`이 화면 출력 + `출력조서/{회사}_실행리포트.txt`
  저장 + **종료 전 항상 창 유지**(exe가 즉시 안 닫히게). LLM은 못 보니 이 리포트가 사용자의 눈.
- **포맷 호환 전략 (구현됨)**: 만능 파서 대신 3겹 안전망. 실제 만난 다양성에만 노력 투입.
  - **C-1차 (오프라인·무료)**: `extractors/_headers.map_headers` — 1차 정확 동의어 매칭(검증됨) +
    2차 **부분일치(fuzzy) 폴백**(못 찾은 *필수* 필드만, 미사용 열 키워드 검색). 회귀 위험 0(폴백 전용).
    ledger·insurance가 사용. 새 파서도 이 헬퍼로 헤더 매핑.
    - **헤더 라벨 검색 = 항상 다중 헤더행 스캔 (필수 — 반복 사고 예방)**: 라벨로 열을 찾을 땐
      **첫 헤더행만 보지 말고 서브헤더행(2~3행 헤더)까지 반드시 함께** 본다. 라벨이 어느 행에
      올지는 회사마다 달라서(같은 'Description'이 회사 A는 1행·회사 B는 서브행) 첫 행만 보면
      **조용히 None → 그 열 데이터 통째 누락**(에러 없는 침묵 손실)이 난다. 라벨 우선 + 위치
      오프셋은 폴백(라벨이 아예 없는 변형 전용 — 병합셀이면 위치가 어긋날 수 있어 라벨이 더 견고).
      (근거: 수정사항집계 'Description'이 서브헤더행에 있어 한 회사 BBDD 각주가 통째 누락된 사건.)
  - **C-2차 (마스킹 LLM, 미해결 필드만)**: `format_adapt.py`. C-1차로도 필수 필드를 못 찾으면 그
    **헤더 행만** 좁게 LLM에 질의. **셀 값은 전부 마스킹**(`#`=자릿수/`○`=문자길이/`∅`=빈칸, 실값 0 누출
    — `mask_cell`), 헤더 라벨(구조 메타)만 평문. 답은 코드가 **인덱스 범위+열 타입 검증** 후 사용.
    해소된 매핑은 **학습 캐시**(`변환자료/learned_headers.json`)에 저장 → `_headers.set_learned`로
    다음 실행부터 오프라인 자동 인식(복리). 파이프라인이 시작 시 캐시를 주입.
  - **보안(엄수)**: API 키는 코드/exe **내장 금지**, 외부 `.env`/환경변수(`ANTHROPIC_API_KEY`)에서만 로드.
    **키 없으면 LLM 비활성=100% 오프라인**(C-1차+학습캐시만). egress 전 **동의(Y/N)+전송내용 표시** 필수.
    ⚠️ 라이브 LLM 자동 트리거를 파서 흐름에 박기 전, **실제 변형 양식 샘플로 프롬프트/응답 검증** 필요
    (현 4개 회사는 전부 C-1차로 매핑됨 → 트리거 발동 사례 없음). 마스킹은 실데이터로 0 누출 검증 완료.

3. **시각 렌더 확인 (생성기 개발 시 — 토큰 효율적 운용)**: 프로그래밍 검사는 "정의한 항목"만 잡는
   프록시라 예상 못 한 깨짐(정렬·균형·색 등)은 못 본다. 더 강력한 건 실제 렌더를 보는 것이고,
   `render_qa.render_sheet_to_pdf`(Excel COM → PDF)로 시트를 PDF 렌더하면 멀티모달로 직접 확인 가능.
   - **★ 조서 양식/산출물 제작·가공 시 실제 시각 QA 필수 (위반 금지 — 사용자 지시):** 조서(개별시트·
     매핑표·가이드 포함)를 만들거나 가공한 뒤 **'완료' 보고 전 반드시 `render_qa`로 편집 시트를 PDF
     렌더해 눈으로 확인**한다. 프로그래밍 검사(validate_grid·check_column_fit·byte비교)는 **프록시일 뿐**,
     실제로 **마커 fill 과다(열 전체 노랑)·가이드 텍스트 넘침/뭉개짐·회사명 등 타사 잔재·정렬·#DIV/0!
     노출** 같은 결함은 **시각으로만** 잡힌다(실측: 프로그래밍 100점인데 P 수금현황 노랑 과다, B 명세서
     타사명 잔재가 렌더에서 발견됨). 다수 조서는 핵심 시트를 일괄 렌더해 전수/표본 확인한다.
   - **언제**: 생성기·템플릿·config를 **신규 작성/수정**할 때 → 그 시트 1장만 렌더해 시각 확인(1회성).
   - **언제 안 함**: 안정화된 생성기를 **새 데이터로 production 실행**할 때는 프로그래밍 검사만
     (데이터 변동 깨짐은 validate_grid/check_column_fit가 잡는 클래스). 배포 exe는 해당 없음(0 토큰).
   - 이유: 데이터 얹기는 사실상 늘 양식에 영향 주므로, 트리거는 "데이터냐"가 아니라 "**로직/템플릿을
     손대는 개발 중이냐**". 시각=로직 1회 검증, 프로그래밍=매 실행 데이터 가드. 크로스체크 루프보다 쌈.

## 핵심 데이터 소스

| 파일 | 역할 |
|---|---|
| **계정별원장_변형** | 계정별 기초·당기증감·기말 잔액 — 조서의 핵심 숫자 |
| **정산표** | 전체 계정 잔액 요약 / 대사 및 완전성 검증 기준 |
| **전자조회서** | 외부 확인 잔액 (현금·차입금 등) — 대사 단락 |
| **분개장_변형** | 계정별 거래 명세 — 필요시 특정 계정 상세 추출 |

회사 제시 원본(회사제시)과 코드 처리용 표준(변형)을 분리한다.
코드는 항상 변형(표준) 파일을 입력으로 받는다.

## 처리 흐름

프로젝트의 두 축: **(1) 비정형 입력의 정형화** + **(2) 정형 데이터의 노가다성 조서 매핑 자동화**.

```
입력: 계정별원장·잔액(기초·증감·기말) + 조회서(외부확인) + 발송 CS  [+ 정산표/분개장(선택)]
   │
   ├─ STEP 1. 데이터 추출·정형화  회사마다 다른 비정형 입력 → '이상적 양식 중간다리'로 정규화
   │                              (헤더 매핑 C-1차/C-2차, 잔액 스냅샷↔거래처원장 라우팅)
   ├─ STEP 2. 조서 초안 생성     [계정유형별 분기] 정형화 데이터를 조서 양식에 매핑 + 합계 자동대사
   ├─ STEP 3. 양식 무결성 검증   격자·소계·열·제목 게이트 (시각 확인의 프로그래밍 대체)
   ├─ STEP 4. 서술 생성         Test 목적·방법·결론 초안 (Claude API, 선택)
   └─ STEP 5. 회계사 보완·완성   초안을 검토하고 추가 절차를 수행해 완성
```

조서 생성 방식: 변형 양식에서 숫자를 제거한 파일을 작업 템플릿으로 사용.
고정 셀 채우기와 동적 행·단락 삽입을 함께 지원한다.

## 계정유형별 처리 분기

조서 양식(틀)은 표준화되어 있으나, 데이터를 채우는 로직은 계정 성격에 따라 다르다.

- **잔액형** (현금, 차입금): 기초 → 증감 → 기말 잔액 중심으로 채움
- **변동분해형** (유형자산): 거래를 취득·처분·감가상각 등 원인별로 분류해 채움

분기는 "어떤 양식을 쓸지"가 아니라 **"계정별로 다른 데이터 채우기 로직"**이 핵심.
처음부터 모든 분기를 구현하지 않고, 계정을 하나씩 추가하며 점진적으로 확장.

## 구현 범위 (우선순위 순)

1. 현금·금융기관 조서 (A-1: 완전성 체크 + 조회서 대사 + 기타조회)
2. 유형자산 조서 (변동분해형 롤포워드)
3. 검토 인터페이스

**접근 방식**: 1개 계정 조서를 end-to-end로 완성한 뒤, 동일 패턴으로 확장.
(각 항목의 진행 상태는 자동 메모리에서 추적 — 이 문서는 우선순위·범위만 정의.)

> **범위 제외 (2026-06):** '감사포인트 매핑(정산표 → 감사 절차 서술)'은 구현 범위에서 뺀다.
> 현재 초점은 **비정형 입력의 정형화 + 정형 데이터의 노가다성 조서 매핑 자동화**이며,

## 양식 보존 공용 도구 (조서 맵핑 공통 핵심)

조서는 계정/구분별 **가변 길이 행 + 섹션마다 소계** 구조라, 지정 계정 이상을 맵핑하면
행수가 템플릿과 달라져 양식(테두리·소계·수식 참조)이 무너진다. 이는 조서 종류와
무관한 **공통 문제**이므로 `generators/base.py`(`WorkpaperGenerator`)에 재사용 도구로 둔다.

- `capture_row_style` / `apply_row_style` — 도너 행 스타일을 캡처·복사 (테두리 보존)
- `clear_region` — 영역 값·스타일·병합 초기화 (동적표용) / `clear_values` — 값만 비움(고정격자 보존용)
- `fit_column_widths` — 열 너비를 내용에 맞게 보정(긴 주소 등 잘림 방지, 한글 전각 2배 계산)
- `render_subtotal_sections(...)` — 가변 섹션을 (데이터 행 + 소계 행)으로 양식 보존하며
  렌더하고 **소계 행번호를 반환**. 상위 합계(예: 모집단 총금액)는 이 행번호로 수식 재계산.
- 셀 주소·열 매핑·각주 규칙은 config YAML / `row_decorator` 로 주입 → 코드 하드코딩 금지.

신규 생성기(A-300, A-0 등)는 이 도구를 **상속해 그대로 사용**한다(별도 양식 분기 로직 작성 금지).

### 양식 무결성 검증 (수식으로 못 잡는 영역)

엑셀 수식은 값만 계산하므로 시각적 양식 붕괴(테두리 소실·소계 어긋남·행 밀림)를
잡지 못한다. `validator.py`가 openpyxl로 **구조 불변식**을 검사하는 게이트를 제공한다
(소계 SUM 범위 일치, 라벨 병합, 데이터 행 테두리 보존, 표 범위 밖 누출, 각주 수 일치 등).
git 훅(데이터 유출 차단)과는 별개의, **생성 직후 양식 무결성 게이트**다.

## 구현 재사용 규칙 (재발명 금지)

새 조서를 만들 때는 **먼저 기존 자산을 재사용**하고, 조서별 차이는 config(YAML)로만 둔다.
양식 분기를 새로 짜지 말고 기존 도구·패턴을 활용한다.

- **★ 완성본은 openpyxl로 열어 저장 금지 (위반 금지 — 핵심 규칙).** 회계법인 완성본 조서는 보조시트에
  양식컨트롤(조서당 ~94개)·매크로(.xlsm)·도형이 가득해, openpyxl로 열고 저장하면 **전부 소실**된다(실측
  94→0). 따라서 완성본은 **절대 openpyxl로 열어 저장하지 않고**, 반드시 **`extract_light`(총괄표 1장만
  가벼운 임시본으로 추출) → 생성기로 채움 → `graft_sheet`(zip/XML 레벨로 그 시트만 무손실 이식)** 패턴을
  쓴다(`generators/sheet_surgery.py`). 다른 시트·컨트롤·매크로는 바이트 그대로 보존된다. graft는 시트별로
  연쇄 가능(1차 출력 → 다음 시트 graft …). **위치는 절대행이 아니라 앵커/라벨로 찾는다**(1차 행삽입으로
  밀려도 견고). 행삽입 생성기는 `base.shift_formula_rows`로 보존영역 수식 참조를 보정한다.
- **추출기**는 모두 **이상적 양식 중간다리**로 수렴시킨다. 헤더 매핑은 공용
  `extractors/_headers.map_headers`(C-1차)를 쓰고, 미해결 필수열만 C-2차(마스킹 LLM)로 폴백한다.
- **★분개장은 raw로 직접 다루지 않는다 (필수 — 도구 재사용·일관성).** 회사제시 분개장은
  `extractors/journal.parse_journal`로 **이상적 분개장 양식**({날짜·번호·계정과목·거래처·적요·차변·대변})
  으로 먼저 변환한다(변환은 **정산표 최초 매핑/조서생성 시 자동** 수행, `변환자료`에 해시캐시).
  **개별 조서(G 유무형자산 등)가 분개장을 활용할 땐 반드시 그 이상적 분개장에서 꺼내 쓴다** — 각 조서가
  raw 시트를 자체 헤더감지로 직접 읽으면 안 됨(중복·포맷취약·캐시 미공유). 도메인 추출기
  (`extractors/fixed_asset_movements.extract_fixed_asset_movements` 등)는 **이상적 라인 리스트를 인자로 받아**
  필터링만 한다. 파이프라인은 `load_with_cache(…, parse_journal, tag="journal…")`로 변환을 분리·공유한다.
- **생성기**는 `generators/base.py`(`WorkpaperGenerator`)를 **상속**해 위 "양식 보존 공용 도구"를 그대로 쓴다.
- **검증기** `validator.py`의 양식 무결성 게이트를 생성 직후 통과시킨다(위 검증 섹션 참조).
- 셀 주소·열 매핑·각주·환율 등 회사/조서별 값은 코드가 아니라 config·런타임 파라미터로 주입한다.

> 구현된 추출기·생성기·검증기·파이프라인의 **실제 목록과 시그니처는 코드(`src/`)가 단일 출처**다.
> 진행 상태·매핑 상세는 자동 메모리(`project-audit-automation`, `project-a1-analysis`)에 있다.
> 이 문서는 **규칙만** 둔다 — 상태·카탈로그는 여기에 중복 기재하지 않는다.

## 사용자 작업 폴더 (배포 = 회사별 워크스페이스 + 2 실행기)

> 동작 플로우·구현 현황의 공개용 서술은 [docs/동작플로우_및_구현현황.md](docs/동작플로우_및_구현현황.md).

배포 패키지는 **실행기 2개**로 동작하며, 한 폴더 안에서 여러 회사를 **회사별 워크스페이스**로
격리 관리한다. 공통 계층은 `src/workspace.py`(`select_workspace`/`workspace_for`/`prompt_company_info`/
`push_active`). 프로그램은 실행파일/스크립트 위치 기준으로 폴더를 참조한다(`A1_WORK_DIR`·`A1_*_DIR`로 재정의 가능).

```
자동화프로그램/
 ├─ 작업생성기.exe   (easy_run_setup) 회사명 입력 → 작업/{회사}/ 4폴더 생성(+입력안내문·환율/상각 시드 동봉)
 ├─ 조서생성기.exe   (easy_run_all)   회사 선택 → 회사정보 1회 → 조서 메뉴(하나만/여러개/전체)로 생성
 ├─ 양식자료/        조서 템플릿 (회사 무관 공유, 시트 구조로 동적 선택)
 └─ 작업/{회사}/     회사명 = 폴더명(자동 인식). 회사정보(기준일·작성자·검토자)는 변환자료/_회사정보.json에 1회 저장→재사용
     ├─ 입력자료/    원본 엑셀 투입
     ├─ 변환자료/    이상적 양식으로 변환·캐시(소스 해시 기반, 회사별 격리)
     ├─ 참고자료/    전기조서·환율·내용연수 (회사별)
     └─ 출력조서/    최종 생성 조서 + 실행 리포트
```

- 통합 실행기는 7개 조서 모듈의 `main()`을 **importlib 동적 로드**로 재사용한다(spec **hiddenimports에
  7개 easy_run 모듈 명시 필수**). 회사·회사정보를 1회 고르면 `push_active`로 각 모듈이 재선택·재입력 없이 재사용.
- `양식자료`·config는 회사 무관 **공유 자산**(dev=`_internal/config`/`_internal/양식`, 배포 .exe=내장).
  `작업/`·5폴더는 실데이터·법인양식을 담으므로 **`.gitignore`에 등록**(공개 금지).
- 레거시 평면 배포(작업/ 도입 전) 호환: 루트 `입력자료/`에 파일이 있으면 `[0] 기존 입력자료 그대로` 폴백 제공.

### 입력 파일명 표준 (단순 역할명 + 레거시 호환)

`입력자료/`의 파일은 **역할 한 단어**로 인식한다(앞에/뒤에 회사명·날짜 붙여도 됨). 작업생성기가
넣어주는 `★먼저읽기_입력파일안내.txt`가 이 표준을 안내한다. `discover_inputs`는 **추가식 매칭**
(단순 역할명 1차 + 레거시 `필수자료N` 토큰 폴백)이라 무회귀로 둘 다 인식한다.

| 역할명(권장) | 역할 | 레거시 토큰 |
|---|---|---|
| `정산표` | 정산표 (대부분 조서 기준) | — |
| `조회서` | 조회서 취합엑셀 | `필수자료2_조회서…` |
| `잔액` | 거래처잔액현황(원장) 또는 결산보고서 (택1) | `필수자료3_잔액_유형1/2…` |
| `분개장` | 분개장 | `필수자료4_분개장…` |
| `발송CS` | 발송 Control Sheet | `필수자료1_발송CS…` |
| `관리대장` | 고정자산 관리대장 (유무형) | — |
| `개별명세_A1` | 개별 잔액명세서(선택) | `보조자료_A1…` |

- **확장 규칙**: 새 조서가 새 입력을 요구하면 역할명을 추가하고 `discover_inputs`/해당 `_find`에 한 줄만
  더한다. **기존 매칭·파서는 건드리지 않는다**(추가식 — 회귀 위험 0).
- **확장자 규칙 (필수)**: 엑셀 입력·조서·양식 파일 탐색은 **`.xlsx`만 보지 말고 `.xlsm`(매크로 포함)·`.xls`도
  반드시 포함**한다. 회계법인 조서는 매크로 양식이 흔해 `.xlsm`이 많다(매출채권·매입채무·매출원가 등
  완성본 조서가 전부 `.xlsm`인 경우가 있음). glob은 `*.xls*` 또는 `("*.xlsx","*.xlsm","*.xls")`로 잡을 것.
  (근거: `**/*.xlsx`만 글로브해 .xlsm 조서를 통째로 누락한 사건.)

## 데이터 정책 (엄수)

- `_internal/` 폴더 내 파일은 절대 공개 저장소에 포함하지 않는다.
- 합성 데이터라도 그 양식 구조가 법인 자산에 해당하므로 제외.
- API 키는 `.env`로만 관리. 코드에 하드코딩 금지.
- 실무 데이터를 코드 예시나 테스트에 직접 사용하지 않는다.
- **자동 게이트**: `.githooks/pre-commit`(활성: `core.hooksPath=.githooks`)이 커밋 시
  `_internal/`·엑셀·비밀파일 경로, 사업자/주민번호 패턴, 회사명(denylist) 유입을 차단한다.
  회사명 목록은 그 자체가 유출이므로 gitignore된 `_internal/security/denylist.txt`에서 읽는다.
  (커밋 게이트일 뿐 — 양식 무결성 게이트 `validator.py`와는 별개.)
